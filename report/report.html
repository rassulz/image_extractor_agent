The problem

You have a folder of photos and only basic metadata — no descriptions, no labels. How do you find “a dog running on grass” among thousands of files? Filenames and EXIF data can't answer that. You need a system that understands the visual content of each image and matches it to a free-text query.

The solution

Encode every image once with CLIP into a 512-dimensional vector that captures its meaning. Encode the user's text query into the same space. The closest vectors (cosine similarity) are the best-matching photos. All retrieval runs 100% locally — no cloud, no API key required.

⛔ Core rule — captions.txt is never used

Flickr8k ships with a captions.txt file of human descriptions. The whole point of this project is to prove the system understands images on its own, so that file is excluded from indexing and retrieval entirely. Search is driven purely by CLIP image↔text similarity. Captions used in the UI (optional) are generated by a vision model, not copied.

Why CLIP works across modalities

CLIP (Contrastive Language–Image Pre-training) is a dual encoder: an image encoder and a text encoder trained together on hundreds of millions of image–caption pairs so that a picture and its description land at nearly the same point in a shared vector space. Because both live in that one space, a text vector can be compared directly against image vectors. We use ViT-B/32 pre-trained on laion2b_s34b_b79k.

Reading the score

Cross-modal cosine similarity is low in absolute terms — a strong match is typically 0.25–0.34; you never see 0.9 across modalities. So the system ranks by score, it never thresholds. The UI maps the raw score to a friendly match %.

Key engineering decisions

DecisionWhy it matters
Filter before rankingFor a query like “outdoor photos from July 2026”, we mask candidates by the date filter first, then rank only the survivors by CLIP similarity — so a metadata filter can never be starved out of the global top-k.
Stale-cache guardOn load, the index is validated against metadata.csv (same ids, same order, same model). A stale cache is rejected instead of silently returning wrong images.
Lazy, optional captionsVision captioning is expensive. Search never calls OpenAI; captions are generated only on demand and cached — keeping search instant and token-free.
Pure image indexOnly CLIP image vectors enter the index. Synthetic metadata has no visual grounding, so it influences results through filters, not through the embeddings.

The product — semantic search

Type any description; get the top matching photos ranked by CLIP similarity, each with a match gauge, date, dimensions, the raw score, and a plain-language explanation of why it ranked. Search is instant because it is a single matrix multiply over the cached index.

Query “dog running on grass” → six correct matches in ~40 ms, scored 0.33–0.34.

The product — the semantic map (Atlas)

Every one of the 3,000 image vectors projected from 512-D CLIP space into 2D with PCA, then auto-clustered (KMeans) and labeled by the closest CLIP concept. Clearly separated neighborhoods — nature, animals, people, sport — show the embedding space really does organize images by meaning, with no supervision.

Auto-discovered clusters: animals · nature · people · children · sport · transport.

Results & limitations

What works well

  • Accurate top-k for concrete visual queries (dogs, water, bikes, crowds).
  • Instant search after a one-time index build (~40 ms/query).
  • Correct metadata filtering (filter-before-rank).
  • Fully offline retrieval; no captions.txt, no cloud dependency.

Honest limitations

  • Demo-scale dataset (3,000 of 8,091 images).
  • ViT-B/32 misses fine detail (exact counts, text in images).
  • Flickr8k has no real location metadata, so location is not used; only a synthetic date demonstrates filtering.
  • Query parsing is rule-based; an LLM parser would handle richer phrasing.
